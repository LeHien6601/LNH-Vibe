# Vision

## Project Name

Game Factory

## Mission

Build a professional AI-assisted game production system that can repeatedly create release-ready casual games with high quality, clear structure, reusable workflows, and fast Unity porting.

The system is not just a codebase. It is a production pipeline that turns a game idea into a complete product:

Idea → Design → Prototype → Assets → Implementation → Testing → Polish → Unity Port → Release

## Long-Term Goal

Create a scalable game production framework capable of supporting many future games while maintaining product quality, consistency, and release readiness.

The final system should allow new games to be created faster over time by reusing:

* design templates
* gameplay modules
* UI patterns
* asset workflows
* level systems
* save systems
* monetization systems
* QA checklists
* Unity porting rules
* AI workflow instructions

## Primary Product Type

The first target product category is casual mobile games, especially:

* puzzle games
* sorting games
* merge games
* idle games
* simple arcade games
* lightweight strategy/puzzle hybrids

The system may support other genres in the future, but the first version should focus on small, polished, release-ready casual games.

## Core Philosophy

The project should not start by building a huge general-purpose engine.

Instead, it should start by shipping one small complete game, then extracting reusable systems from real production needs.

Build product first.
Refactor into modules after proven reuse.
Scale only after repeated success.

## What This System Should Produce

Each game created by the system should aim to include:

* clear core loop
* complete playable prototype
* polished user interface
* consistent art direction
* level/content structure
* save/load support
* audio feedback
* monetization support when needed
* release checklist
* store listing assets
* Unity porting plan
* documentation for future reuse

## What This System Is Not

This project is not intended to be:

* a massive all-purpose game engine
* a random collection of prompts
* a one-off prototype generator
* a fully automated game creator with no human review
* a system where AI makes major product decisions without approval
* a project that sacrifices quality for speed

## Human Role

The human developer is the final decision maker.

The human is responsible for:

* choosing product direction
* approving gameplay feel
* reviewing creative direction
* deciding whether a game is release-ready
* validating quality
* publishing the final product

AI tools assist the process, but they do not replace product judgment.

## ChatGPT Role

ChatGPT is used for:

* strategy
* system design
* game design
* architecture discussion
* documentation
* task planning
* review
* asset prompting
* creative direction
* Codex task preparation

ChatGPT should focus on the big picture and reusable knowledge.

## Codex Role

Codex is used for:

* implementation
* file editing
* refactoring
* test execution
* project setup
* documentation updates
* Git commits
* GitHub push or pull request workflow when configured

Codex should work from clear tasks and should not invent large product direction without approval.

## Documentation Role

Documentation is the single source of truth for the project.

The project should describe itself through Markdown files so that ChatGPT, Codex, and future AI tools can understand the current state without reading the entire source code every time.

Key documents should include:

* vision
* north star
* architecture
* current scope
* module index
* decisions
* workflow state
* changelog
* release checklist

## Success Definition

This project is successful when it can repeatedly create small but complete games faster than building each one from scratch, while preserving quality, maintainability, and release readiness.

The first major success milestone is not “support thousands of games.”

The first major success milestone is:

Create one complete release-ready game using the pipeline.

The second major success milestone is:

Create a second game faster by reusing parts of the first game’s workflow and modules.

The third major success milestone is:

Create a reusable production system proven by multiple shipped or release-ready games.
